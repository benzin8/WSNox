# Переезд на новый сервер и домен

Чеклист миграции WSNox со старого VPS (84.23.51.38, `wsnox.urldot.ru`) на новый сервер и новый домен.

## Что уже сделано

- [x] Бэкап Postgres снят: `wsnox_20260713.dump` (pg_dump, custom format, ~130 КБ)
- [x] `.env` прода сохранён как `prod.env` (лежат локально, в репозиторий не коммитить!)
- [x] Написан `scripts/bootstrap-new-server.sh` — развёртывание с нуля на новом сервере
- [x] В `deploy.yml` добавлен `FRONTEND_BASE_URL` (домен больше не захардкожен только в коде)

## Что переезжает, а что нет

| Компонент | Действие |
|---|---|
| Postgres (пользователи, чаты, сообщения) | Восстановить из дампа |
| Медиа (голосовые, фото, видео) | Ничего — лежат в S3 (Yandex Cloud, bucket `wsnox-media`), сервер их не хранит |
| Redis | Не переносим — кэш/эфемерное, поднимется пустым |
| SSL-сертификаты | Не переносим — выпускаются заново под новый домен |
| Docker-образ | Не переносим — тянется из GHCR |

## Порядок действий

### 1. DNS
На новом домене создать A-запись на IP нового сервера. Дождаться резолва (`nslookup домен`).

### 2. Новый сервер
```bash
# Скопировать на новый сервер бэкапы:
scp prod.env user@NEW_IP:~/WSNox/.env      # каталог создать заранее или после clone
scp wsnox_20260713.dump user@NEW_IP:~/

# На сервере:
DOMAIN=новый.домен EMAIL=почта bash scripts/bootstrap-new-server.sh
```
Скрипт ставит Docker, клонит репозиторий, патчит nginx-конфиг, выпускает сертификат Let's Encrypt (двухфазно), поднимает стек и восстанавливает базу из дампа.

В `.env` добавить строку `FRONTEND_BASE_URL=https://новый.домен`.

### 3. Замена домена в коде (коммит в репозиторий)
Старый домен зашит в этих файлах — заменить на новый:

- `src/messenger/backend/core/config.py:44` — дефолт `frontend_base_url`
- `src/messenger/frontend_react/index.html` — canonical, og:url, og:image, twitter:image
- `src/messenger/frontend_react/public/robots.txt` — ссылка на sitemap
- `src/messenger/frontend_react/public/sitemap.xml` — оба `<loc>`
- `nginx/wsnox.conf` — `server_name` (×2) и пути к сертификатам (×2)
- `tests/test_webauthn.py`, доки — по вкусу

### 4. GitHub Actions secrets (Settings → Secrets → Actions)
Обновить:
- `SERVER_HOST` → IP нового сервера
- `SERVER_USER` → пользователь на новом сервере
- `SERVER_SSH_KEY` → приватный ключ (сгенерировать пару, публичный — в `~/.ssh/authorized_keys` на новом сервере)
- `FRONTEND_BASE_URL` → `https://новый.домен` (новый секрет)

Остальные секреты (DB_*, SMTP_*, SECRET_KEY, VAPID_*, S3_*) остаются как были.

### 5. Проверка
- Пуш в `main` (или ручной запуск Deploy) — CI должен задеплоить на новый сервер
- Открыть `https://новый.домен` — логин, отправка сообщения, загрузка фото (проверка S3), голосовое
- Проверить выпуск/автопродление сертификата: `docker logs messenger_certbot`

### 6. Гашение старого сервера
Только после того, как новый живёт стабильно:
```bash
cd ~/WSNox && docker compose -f docker-compose.prod.yml down -v
rm -rf ~/WSNox ~/wsnox_*.dump
```

## Известные последствия смены домена

- **Биометрия (WebAuthn/passkeys) слетит у всех** — credential привязан к rp_id (домену). Пользователи заново включат биометрию на новом домене. Перенести нельзя, это свойство стандарта.
- **Push-подписки слетят** — подписка привязана к origin. Service worker переподпишет пользователей при первом заходе на новый домен (VAPID-ключи менять не нужно).
- **Сессии/куки слетят** — все перелогинятся один раз.
- Старые ссылки-приглашения `https://старый.домен/join/...` перестанут работать.
