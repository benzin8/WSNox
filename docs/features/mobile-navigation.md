# Мобильная навигация

На мобильных (`<md:`) список чатов и открытый чат живут в одном горизонтальном «слайдере», который переключается между двумя состояниями. На десктопе тот же контейнер раскрывается в обычный flex-layout — без дублирования разметки.

## Архитектура

```
<div className="relative flex-1 overflow-hidden md:flex">
  <div ref={sliderRef}
       className="absolute inset-y-0 left-0 flex md:contents"
       style={{
         width: '200%',
         transform: `translateX(${mobileTranslateX})`,
         transition: ...
       }}>
    <Sidebar className="w-1/2 md:w-80 md:flex-shrink-0" />
    <ChatPanel className="w-1/2 md:flex-1 md:w-auto" />
  </div>
</div>
```

Ключ к десктопу — `md:contents`: слайдер-обёртка перестаёт генерировать собственный box, дети становятся прямыми детьми flex-родителя и ведут себя как обычные колонки. Никакого CSS-дубля и conditional rendering.

## Слайд

| Состояние | `mobileTranslateX` | Что видно |
|-----------|---------------------|-----------|
| `mobileView === 'list'` | `0%` | список чатов |
| `mobileView === 'chat'` | `-50%` (от ширины слайдера) | открытый чат |

Слайдер сам 200% ширины родителя, две панели по 50% слайдера = по 100vw каждая. Сдвиг ровно на 50% слайдера = на 100vw — ровно одна панель уезжает за край, появляется вторая.

Анимация: 300 мс `cubic-bezier(.4,0,.2,1)`. Включается только в момент смены `mobileView` (через короткий `isAnimating` flag), чтобы не мешать жесту. `prefers-reduced-motion` → анимация мгновенная.

## Edge-swipe back

Жест в стиле iOS: тянем слева направо в первых 24px экрана → закрываем чат. Реализовано в `useEdgeSwipe`:

1. `touchstart` — фиксируем точку только если `clientX <= edgeZone` (24px).
2. `touchmove` — определяем направление после 10px движения. Если вертикаль доминирует → отменяем жест (это скролл сообщений). Иначе шлём `progress = dx / window.innerWidth` в `onDrag` и `preventDefault` для блокировки скролла.
3. `touchend` — если протянул > 30% экрана **или** скорость > 0.4 px/мс → `onSwipeComplete()` (закрыть чат). Иначе `onSwipeCancel()` (откатить).

Во время жеста transition выключается → палец «прилипает» к слайдеру. После завершения transition снова включается — пружинистое доезжание.

| Параметр | Значение | Что значит |
|----------|----------|------------|
| `edgeZone` | 24 px | зона активации от левого края |
| `threshold` | 0.3 | минимум 30% ширины экрана |
| `velocityThreshold` | 0.4 | px/мс — быстрый флик закрывает с меньшим протягиванием |

## Подводные камни (на чём ловили)

- **Чёрный экран при открытии чата.** Изначально слайдер двигался на `translateX(-100%)`, что = -200vw, и обе панели уезжали полностью за экран. Правильное смещение — `-50%` от собственной ширины слайдера. Если будешь рефакторить — помни, что 100% слайдера ≠ ширина экрана.
- **Жест с половинной чувствительностью.** `useEdgeSwipe` считал прогресс относительно `containerRef.offsetWidth`, а это слайдер шириной 2× экрана. Перевели на `window.innerWidth` — теперь threshold 0.3 действительно означает 30% экрана, а не 60%.
- **«Перекидывает» в чат + кривая вёрстка, когда пришло сообщение во время просмотра списка.** На back-навигации `mobileView` менялся на `'list'`, но `activeChat` оставался выставленным. Когда собеседник присылал сообщение в этот чат: (1) `setMessages` пушил его в state скрытой панели, (2) effect авто-скролла дёргал `messagesEndRef.scrollIntoView()` на off-screen элементе — iOS Safari подтаскивал layout viewport вбок, ломая transform слайдера, (3) `markChatAsRead` ошибочно срабатывал, `unread_count` не инкрементировался. Фикс — ввели `viewingChatIdRef` рядом с `activeChatIdRef` (см. ниже) и гейтнули им сайд-эффекты + сам `scrollIntoView`.
- **`position: fixed` ловит transform слайдера, оверлеи едут вбок.** `MessageActionMenu` использовал `fixed inset-0` для full-viewport бэкдропа. CSS spec: ancestor с `transform` создаёт containing block для fixed-потомков. Наш слайдер шириной 200% c `translateX(...)` — именно такой ancestor, поэтому `inset-0` ровнялся не по экрану, а по слайдеру, и центр уезжал на пол-экрана. Фикс — рендерить такие оверлеи через `createPortal(..., document.body)`, чтобы они жили вне трансформированного предка. Касается любого нового full-screen modal/popover — портал обязателен.

## Active vs viewing chat

Два ref-а в `ChatPage.jsx`, формально похожих, но семантически разных:

| Ref | Когда выставлен | Кто читает |
|-----|-----------------|------------|
| `activeChatIdRef` | всегда = `activeChat?.id` | `useChatSocket` (накопление сообщений в state, чтобы при возврате к чату ничего не было потеряно), reconnect-refetch |
| `viewingChatIdRef` | `activeChat?.id` если десктоп **или** `mobileView === 'chat'`, иначе `null` | mark-as-read, инкремент unread, подавление нотификаций, авто-скролл |

Правило: если фича про **«пользователь сейчас реально смотрит в чат»** — гейти через `viewingChatIdRef`. Если про **«в state-е лежат сообщения этого чата»** (накопление, синхронизация) — `activeChatIdRef`. Сервер использует ту же логику (`viewing_chat` событие тоже учитывает `mobileView` — `ChatPage.jsx:180`).

## Ограничения

- Нет swipe-влево из списка → последний активный чат. Опционально, можно добавить зеркалированием `useEdgeSwipe`.
- Edge-зона жёсткая (24px). Под iPhone с очень узкими рамками может конфликтовать с системным back-gesture в браузере — решение, если поплывёт: переключиться на жест анивно по всей ширине + детект horizontal-only с тайм-аутом.
