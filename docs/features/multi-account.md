# Мультиаккаунтинг

Вход под несколькими аккаунтами одновременно и переключение между ними в один клик из модалки профиля. На экране выбора виден счётчик непрочитанного по каждому аккаунту.

## Что входит

| Возможность | Где |
|------|------|
| Хранение нескольких аккаунтов + активный | `features/accounts/accountStore.js` |
| Переключение аккаунта (reload) | `accountStore.switchAccount` |
| Добавление аккаунта через обычный вход, не затирая текущий | `pages/auth/LoginPage.jsx`, `RegisterPage.jsx` + `accountStore.upsertAccount` |
| Обход редиректа auth-страниц, пока добавляем аккаунт | `App.jsx` — `PublicOnlyRoute` + `isAddingAccount()` |
| Блок «Аккаунты» в профиле (аватар, имя, бейдж, галочка активного, выход, «+добавить») | `features/accounts/AccountsBlock.jsx`, встроен в `components/profile/ProfileModal.jsx` |
| Счётчик непрочитанного по каждому аккаунту | `features/accounts/useAccounts.js` → `GET /chats/unread-total` |
| Logout через стор (переключение на следующий / выход) | `pages/chat/ChatPage.jsx` — `handleLogout` |
| Сумма непрочитанного для бейджа | `GET /chats/unread-total` (`ChatCRUD.get_unread_total`) |
| Обновление пары токенов по refresh-токену | `POST /auth/refresh` (`decode_token`) |

## Архитектура

Вся логика мультиаккаунта — на клиенте. Бэкенд получает два самостоятельных эндпоинта, полезных и вне фичи.

### Хранилище (`localStorage`)

```text
accounts          — [{ user_id, display_name, avatar_url,
                       access_token, refresh_token, needs_login }]
active_account_id — user_id активного аккаунта
adding_account    — флаг «идёт добавление аккаунта» (временный)
```

**Совместимость:** легаси-ключи `access_token` / `refresh_token` всегда держатся равными токенам активного аккаунта. Поэтому весь существующий код (а он повсюду читает `localStorage.getItem('access_token')` напрямую) работает без изменений — единый источник правды инкапсулирован в `accountStore`, наружу торчат привычные ключи.

### Переключение

`switchAccount(userId)` пишет токены выбранного аккаунта в легаси-ключи, ставит `active_account_id` и делает `window.location.reload()`. Reload выбран сознательно: WS-сокет, список чатов и контексты (notifications, presence, theme) проще переподнять с нуля, чем рефакторить глобальный auth-стейт, которого в проекте нет.

### Добавление аккаунта

«+ Добавить аккаунт» ставит флаг `adding_account` и ведёт на `/auth/send-code`. Пока флаг стоит, `PublicOnlyRoute` пускает на auth-страницы несмотря на активный токен. При успешном входе/регистрации ответ не перезаписывает текущий аккаунт, а добавляется в `accounts` (`upsertAccount`), после чего происходит переключение на новый.

### Счётчик непрочитанного

`useAccounts` при открытии профиля параллельно запрашивает `GET /chats/unread-total` токеном каждого аккаунта и кладёт числа в бейджи. Ошибка/`401` по одному аккаунту — просто без бейджа, остальной список не падает.

### Миграция старых сессий

Сессия, залогиненная **до** появления мультиаккаунта, имела только легаси-токены и не была в `accounts`. Чтобы она не терялась при добавлении второго аккаунта, на загрузке чата (`ChatPage`) вызывается `seedCurrentAccount(...)` — текущий пользователь заносится в стор по живым легаси-токенам и помечается активным. No-op, если активный аккаунт уже в сторе.

## Бэкенд

### `GET /chats/unread-total`

Auth: `get_current_user`. Возвращает `{ "unread_total": <int> }` — сумму непрочитанного по всем чатам юзера (private: `is_read=False` для адресованных мне; group: сообщения от других без строки `MessageRead(me)` — те же правила, что в `GET /chats/`).

### `POST /auth/refresh`

Тело `{ "refresh_token": "<...>" }`. Валидирует refresh-токен (`type == "refresh"`, подпись, срок) через чистый `decode_token`, проверяет существование юзера, возвращает новую пару токенов. На любой сбой — `401`. Полезен и вне мультиаккаунта: любая сессия может пережить протухание access вместо жёсткого разлогина.

## Вне MVP

- Фоновые пуши/WS по неактивным аккаунтам (real-time и пуши — только по активному).
- Серверная привязка нескольких аккаунтов к одному устройству/владельцу.

## Лимиты

- До 5 аккаунтов (`MAX_ACCOUNTS` в `accountStore.js`).
