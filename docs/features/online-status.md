# Онлайн-статус (real-time presence)

## Описание

Каждый пользователь имеет онлайн-статус, который вычисляется автоматически по факту активного WebSocket-подключения. Зелёная точка на аватаре в списке чатов и надпись «в сети» / «не в сети» в шапке чата отражают реальное присутствие собеседника.

## Как определяется онлайн

Источник правды — ключ в Redis `presence:{user_id}` со сроком жизни (TTL) 60 секунд.

- При подключении к WebSocket сервер ставит ключ `presence:{user_id}` с TTL 60 с.
- Клиент каждые 30 секунд шлёт `{"type": "ping"}` — сервер продлевает TTL.
- Пока вкладка активна (Page Visibility API) — heartbeat идёт, ключ живёт, пользователь online.
- Вкладка ушла в фон / закрылась / пропала сеть — heartbeat прекращается, ключ протухает за 60 с, пользователь становится offline.
- При корректном закрытии вкладки сервер сразу удаляет ключ (не дожидаясь TTL).

Фоновый **sweeper** раз в 10 секунд проверяет, у кого из подключённых пользователей протух ключ, и рассылает событие offline. Мёртвые сокеты он не закрывает — они могут быть живы (вкладка просто в фоне).

Worst-case задержка перехода в offline при тихом обрыве связи — около 70 секунд (60 с TTL + 10 с интервал sweeper).

## Ручная настройка видимости

В профиле есть поле `presence_preference` с тремя значениями:

| Значение     | Поведение                                                                 |
|--------------|---------------------------------------------------------------------------|
| `null` (Обычный)   | Статус определяется автоматически.                                  |
| `dnd` (Не беспокоить) | Пользователь виден как online, рядом — значок «не беспокоить».    |
| `invisible` (Невидимка) | Для всех остальных пользователь всегда выглядит offline, хотя реально подключён. |

Невидимка скрыта полностью: в ответе API `presence_preference` для невидимки маскируется в `null` для других пользователей — собеседник не видит сам факт настройки. Сам пользователь в `/profiles/me` видит своё реальное значение и всегда `online: true`.

## WebSocket-протокол

Сообщения от клиента:

```json
{ "type": "ping" }
```

Heartbeat каждые 30 с, пока вкладка активна. Других presence-сообщений клиент не шлёт — уход в фон обрабатывается на клиенте (просто прекращается ping).

Сообщения от сервера:

```json
{ "type": "presence", "user_id": 42, "online": true }
```

Рассылаются только тем пользователям, у кого `user_id` есть в общих чатах. Канал доставки — Redis pub/sub `presence_events`, отдельный listener фанит события по локальным сокетам.

## API

| Метод | Путь                  | Описание                                                        |
|-------|-----------------------|-----------------------------------------------------------------|
| GET   | `/chats/presence`     | Батч-снимок: `{"online_user_ids": [...]}` — собеседники, которые сейчас online и не невидимки. |
| GET   | `/profiles/{user_id}` | Поле `online: bool` в ответе, с учётом маски невидимки.          |
| GET   | `/profiles/me`        | `online` всегда `true`, `presence_preference` — реальное значение. |
| PUT   | `/profiles/me`        | Принимает `presence_preference` (`null` / `"dnd"` / `"invisible"`). |

Фронт при открытии списка чатов делает один запрос `GET /chats/presence` для начального снимка, затем поддерживает состояние через WebSocket-события `presence`.

## Несколько вкладок

`ConnectionManager` хранит для каждого пользователя множество сокетов (`Set[WebSocket]`). Пока хотя бы одна активная вкладка шлёт heartbeat — общий ключ `presence:{user_id}` жив. Ключ удаляется только когда закрывается последний сокет.

## Известные ограничения

- Нет «был(а) недавно» — только бинарное online / offline.
- Нет push-уведомлений при появлении собеседника в сети.
- Нет лимита на число WebSocket-подключений с одного IP.
