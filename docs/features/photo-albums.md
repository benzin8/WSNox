# Мульти-фото альбомы (коллажи)

Выбрать несколько фото за раз, посмотреть их в лотке-превью (убрать лишнее,
поменять порядок, одна подпись), отправить как один **альбом** — в чате он
рендерится как **коллаж** в духе Telegram.

**v1:** только фото, до 10 штук, одна подпись на альбом.

## Модель данных — сгруппированные сообщения

Каждое фото альбома — **отдельная строка `Message`** с обычными полями вложения,
плюс общий идентификатор группы. Группировка в коллаж живёт только на слое
рендера, поэтому галерея медиа, поиск, очистка S3 и оптимистичный аплоад
переиспользуются без изменений.

- `messages.album_id` (`String(32)`, nullable, indexed) — id группы,
  **генерируется клиентом** (`a<base36 time><rand>`), шлётся с каждым фото пачки.
  `NULL` для одиночных медиа и текста. Миграция `f1a2b3c4d5e6`.
- Порядок — `attachment_meta["album_index"]` (0..N-1), в JSONB (без отдельной
  колонки): нужен только для рендера, задаётся клиентом (параллельная загрузка
  завершается не по порядку).
- Подпись альбома хранится на фото с `album_index == 0`; у остальных пустая.
- Кап (≤10) и формат `album_id` валидируются на сервере
  (`core/albums.py`: `is_valid_album_id`, `is_valid_album_index`).

## Отправка

- Пикер (`AttachmentPicker`) — `multiple`. 2+ фото открывают лоток
  `MediaAlbumComposer`; 1 файл идёт обычным путём (preview-модалка). Видео в
  альбом в v1 не берём — мульти-пик с видео шлёт первый файл одиночно.
- Лоток: миниатюры, крестик «убрать», стрелки ◄ ► для порядка, один инпут
  подписи, «Отправить». Кап 10, лишние отсекаются.
- Транспорт: тот же `POST /chats/{chat_id}/media` **по запросу на фото**, с
  доп. form-полями `album_id` и `album_index`. Клиент шлёт пачку с
  конкуррентностью 3, у всех общий `album_id`; подпись — только с `index 0`.
- Оптимистично: одно сообщение на фото (общий `album_id` + `album_index` в meta),
  поэтому коллаж виден сразу с прогрессом по плиткам; плитки заменяются реальными
  URL по мере ответов / `media_ack`.

## Рендер

- `MessageList` склеивает подряд идущие сообщения с одинаковым `album_id` и
  направлением в один синтетический элемент → `AlbumMessage`.
- Раскладка — чистая функция от числа фото (`albumLayout.js`, юнит-тест
  `albumLayout.test.js`):
  - **2** — два кадра рядом;
  - **3** — широкий сверху + два квадрата снизу;
  - **4** — 2×2;
  - **5–10** — сетка по 3, плитки-квадраты.
  Все плитки квадратные (кроме широкой верхней у 3 фото), поэтому ряды всегда
  выровнены без полного мозаичного алгоритма ТГ.
- Подпись рендерится обычным путём пузыря (текст под коллажем).
- Фуллскрин: тап по плитке открывает `MediaLightbox`, листание по альбому
  стрелками / клавишами ←→ / свайпом, счётчик «i / N».

## Взаимодействие с остальным

| Фича | Поведение |
|---|---|
| Галерея медиа | Каждое фото — отдельное image-сообщение → попадает в галерею как есть. Без изменений. |
| Поиск | Подпись на фото `index 0`; поиск по дате/словам не меняется. |
| Удаление | Удаление любой плитки сносит **весь** альбом (все строки с этим `album_id`) + их объекты S3; `message_deleted` несёт `message_ids` со всеми id. |
| Уведомления | Дедуп по `album_id`: ding/пуш только на первое фото («📷 Альбом»), остальные молчат (`shouldNotify` + `_fanout_offline_pushes`). |
| Превью в списке чатов | «📷 Альбом» (или подпись). |
| Реакции / ответ | На альбом целиком (якорь — первое фото). |
| Каналы | Альбом постит владелец тем же путём. |

## Вне v1

- Видео в альбоме (смешанные фото+видео).
- Поштучное удаление одной плитки.
- Группировка альбома как единого элемента в галерее.
- Полный мозаичный алгоритм ТГ для 5–10.
