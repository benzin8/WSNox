# Group chats — roadmap

MVP описан в **[group-chats.md](./group-chats.md)**. Этот документ — следующие итерации в приоритетном порядке. Каждая итерация — отдельная feature-ветка.

## Итерация 2: «seen by N/M»

**Зачем**: сейчас read receipts в группе скрыты целиком. Хочется лёгкий индикатор «прочитано 3 из 5» без выкатывания полного списка.

**Что**:
- При `message_read` в WS-роутере записывать `MessageRead(message_id, user_id, read_at)` для каждого нового прочитанного.
- В `GET /chats/{id}/messages` для группы возвращать `seen_count` (count distinct user_id) на каждое сообщение отправителя.
- На стороне отправителя — компонент `MessageStatus` для группы показывает «N» вместо «прочитано/отправлено».

**Риски**: вставка `MessageRead` на каждое прочитанное может породить много записей при истории. Решение — батчем (как сейчас `mark_as_read_up_to`).

## Итерация 3: переименование и аватар группы

**Что**:
- `PATCH /chats/{id}` — `{name?, avatar_key?}`. Только админ.
- Поле `Chat.avatar_key` (новая колонка, миграция).
- Frontend: пункт меню «Изменить название», кнопка смены аватара в `GroupInfoModal`. Avatar uploader переиспользует пайплайн `services/media.py::process_image` (тот же resize/JPEG 320 thumb).
- WS-событие `group_updated` для fan-out изменения по `member_ids`.

**Связь с MVP**: `GroupAvatar` сейчас рендерит инициалы. Когда появится `avatar_key`, тот же компонент должен сначала пытаться отдать загруженный thumb, фолбэчиться на инициалы.

## Итерация 4: удаление участника админом

**Что**:
- `DELETE /chats/{id}/members/{user_id}` — только админ, нельзя удалить себя (есть `/leave`), нельзя удалить последнего админа.
- WS-событие `group_member_removed`, удаляемый получает явное «вас удалили из группы» (toast/уведомление).
- В `GroupInfoModal` появляется крестик рядом с участником у админа.

**Краевой случай**: удалённый юзер не должен получать события из этого чата → `chat_events_listener` проверяет, что пользователь всё ещё в `member_ids`.

## Итерация 5: повышение/понижение роли

**Что**:
- `PATCH /chats/{id}/members/{user_id}` — `{role: "admin"|"member"}`. Только текущий админ.
- В `GroupInfoModal` контекстное меню на участнике: «Сделать админом».
- Инвариант: всегда минимум один админ в группе. Понижение последнего — 400.

## Итерация 6: @mentions

**Что**:
- В `InputArea` autocomplete после `@` — выпадающий список участников, фильтр по display_name/username.
- В отправке сохранять `mentions: [user_id]` (новое поле в payload и `message`-таблице как JSONB или separate `message_mention` join-table).
- На приёме у упомянутого пользователя — повышенный приоритет push (`push_priority="high"`) и заметная подсветка в чате.
- Превью в списке чатов: «@you Иван: текст» отдельным стилем.

**Решение по хранению**: отдельная таблица `message_mention(message_id, user_id)` — позволяет джойнить «непрочитанные mentions» в `unread_count` без парсинга JSON.

## Итерация 7: pinned messages

**Что**:
- `POST /chats/{id}/pins/{message_id}` — админ. `DELETE` тот же путь. Лимит N=3 на чат.
- Колонка `Chat.pinned_message_ids: ARRAY[int]` (Postgres) или `Pin` join-таблица — выбрать при дизайне.
- В верхней части `MessageList` баннер с одним закреплённым (кликом — открыть остальные).
- WS-событие `chat_pins_updated` для синхронизации.

## Итерация 8: правильный `unread_count` в группах

**Что**: переписать `unread_group` подзапрос в `ChatCRUD.get_chats` так, чтобы он считал «сообщения с `created_at > last_read_at` пользователя в этом чате». Это требует таблицы `chat_member_state(chat_id, user_id, last_read_at)` или индекса `message_read` по `user_id`.

**Почему отдельно**: текущий подзапрос корректен только когда `MessageRead` начнёт писаться (итерация 2). До этого числа в groups завышены — но это видно только если в группе уже шла активность.

## Что точно НЕ войдёт в roadmap

- **Звонки в группе** — отдельная вертикаль (`Phone`-кнопка в шапке группы скрыта в MVP осознанно).
- **Стикеры/реакции** — общая фича чата, не специфичная для групп.
- **Полнотекстовый поиск по чату** — отдельный проект (нужен Postgres FTS или Meilisearch).
- **Каналы (broadcast)** — другая модель прав, отдельный `chat_type="channel"`.

## Технический долг, который накопил MVP

1. `recipient_id` в `message` остался ради back-compat. Когда все client-сборки обновятся на `recipient_ids`, колонку можно дропнуть (новая миграция). До этого — двойная запись для private.
2. `chat_info.recipient` в WS payload для group равно `None`. Старые клиенты не падают, но в чужих сборках стоит проверить, что они переживают `recipient: null` без TypeError.
3. `should_expose_read_receipts` всё ещё специфичен для private. После итерации 2 он либо переедет в чисто-private код-путь, либо обобщится «expose всем участникам с включёнными receipts».
4. `unread_count` имеет два разных подзапроса (`unread_private`, `unread_group`) которые `COALESCE`ятся. После итерации 8 их можно унифицировать.
