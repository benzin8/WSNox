# Модерация и безопасность пользователей

Набор фич, чтобы WSNox можно было открывать для реальных людей: защита от
навязчивых сообщений, блокировка нежелательных собеседников и бан нарушителей.
Три уровня — **согласие на переписку**, **блокировка** (на стороне юзера) и
**бан** (на стороне модерации).

Это про продуктовое поведение. Технические меры против абьюза (rate-limit,
лимиты загрузок, hardening) описаны отдельно и поверхностно в
[abuse-prevention.md](../security/abuse-prevention.md).

## 1. Согласие на переписку (chat request)

Раньше любой мог написать кому угодно сколько угодно. Теперь первый контакт —
по **обоюдному согласию**, в духе «request to chat».

### Поведение
- Когда вы пишете человеку, с которым ещё не общались, создаётся **чат-запрос**.
  Вы можете отправить **только одно сообщение** — дальше инпут блокируется с
  подписью «Запрос отправлен — продолжить можно будет после того, как его примут».
- У получателя сверху чата — табличка с тремя действиями:
  **Принять** · **Отклонить** · **Спам**.
  - **Принять** — чат становится обычным, оба пишут свободно.
  - **Отклонить** — запрос убирается; история не разворачивается в переписку.
  - **Спам** — то же, что отклонить, плюс сигнал для модерации.
- Инициатор не видит, прочитано ли его сообщение, пока запрос не принят.

### Данные
- `chats.is_request` (Boolean, default `false`) — чат в состоянии запроса.
- `chats.initiator_id` (Integer, nullable) — кто инициировал.
- `create_private_chat` ставит `is_request=True, initiator_id=<current_user>`.
- Лимит «одно сообщение» проверяется на WS-пути отправки:
  `MessageCRUD.count_in_chat_by_sender(...) >= 1` для инициатора, пока
  `is_request` истинно. Ответ получателя снимает флаг (`is_request=False`).
- Миграция `e1f7a93c4d28_add_chat_request_consent`.

### API
| Метод | Назначение |
|---|---|
| `POST /chats/{chat_id}/accept` | принять запрос (только получатель) |
| `POST /chats/{chat_id}/decline` | отклонить запрос |
| `POST /chats/{chat_id}/report-spam` | отклонить + пометить как спам |

Все три защищены guard'ом «только получатель запроса» — инициатор не может
принять собственный запрос.

## 2. Блокировка пользователя

Любой юзер может заблокировать любого другого. Блокировка **взаимно**
обрывает доставку: ни вы заблокированному, ни он вам.

### Поведение
- Кнопка «Заблокировать» — в карточке собеседника (`ChatInfoModal`).
- После блокировки сообщения в обе стороны не доставляются (на WS-пути отправка
  тихо отбрасывается, если есть блокировка в любую сторону).
- Создать новый личный чат с заблокированным тоже нельзя (`get-or-create`
  возвращает `403`, если между вами стоит блокировка).
- Блокировку можно снять.

### Данные
- Модель `UserBlock` (`models/user_block.py`): композитный PK
  `(blocker_id, blocked_id)` + `created_at`.
- CRUD `BlockCRUD` (`crud/block.py`): `block`, `unblock`, `has_blocked`,
  `is_blocked_either` (проверка блокировки в любую из сторон — её и зовёт
  WS-путь и `get-or-create`).
- Миграция `d3a8f6c1b920_add_user_blocks`.

### API
| Метод | Назначение |
|---|---|
| `POST /chats/users/{user_id}/block` | заблокировать |
| `POST /chats/users/{user_id}/unblock` | разблокировать |
| `GET /chats/users/{user_id}/block-status` | статус (заблокировал ли я / меня) |

## 3. Бан пользователя (модерация)

Жёсткая мера: бан выключает аккаунт целиком. В отличие от блокировки (личное
дело двух людей) — это действие **модерации**, доступное по праву.

### Поведение
- Забаненный не может залогиниться (`login` → `403`), а активные сессии и
  WebSocket-сокеты **разрываются на лету**: при следующем действии бан-флаг
  проверяется по кэшу пользователя, и сокет закрывается.
- Бан ставится из админки (`/dashboard` → список юзеров → кнопка бана) с
  подтверждением через `BanConfirmModal` и **указанием причины**.
- Защита от ошибок и эскалации прав:
  - требуется ввести **email** банимого (`confirm_email`) — нельзя забанить не
    того, кого видишь;
  - **проверка ранга** (`role_rank`) — нельзя забанить равного или старшего по
    роли (модератор не банит админа/владельца).

### Данные
- `users.is_banned` (Boolean, server_default `false`),
  `users.banned_at` (timestamptz), `users.ban_reason` (String 300).
- `CachedUser.is_banned` добавлено в снапшот в `core/identity.py` — поэтому
  бан виден на горячем пути auth без похода в БД; при бане кэш `user_auth`
  сбрасывается (`invalidate`), чтобы флаг применился сразу.
- Право `PERM_BAN_USER` (`core/permissions.py`) выдано ролям admin / owner.
- Миграция `c7b1e9a4f206_add_user_ban_fields`.

### Где проверяется бан
- `get_current_user` (HTTP-зависимость) → `403` забаненному.
- `auth/login` → `403`.
- WS `_authenticate` → отказ в установке соединения.
- WS message-handler → закрытие уже открытого сокета (кик «на лету»).

### API
| Метод | Назначение |
|---|---|
| `PATCH /api/admin/users/{user_id}/ban` | `{confirm_email, banned, reason}` → бан/разбан, гейт `PERM_BAN_USER` |

Список юзеров в админке (`GET /api/admin/users`) теперь отдаёт
`is_banned / banned_at / ban_reason`, а в UI рисуется бейдж «забанен».

## Как это связано

| Сценарий | Инструмент |
|---|---|
| Незнакомец пишет рекламу | согласие на переписку — у вас 1 сообщение и кнопка «Спам» |
| Конкретный человек надоел | блокировка (взаимный обрыв доставки) |
| Системный нарушитель / бот | бан из админки (с причиной и проверкой ранга) |

Реактивную защиту от объёмного абьюза (массовые рассылки, спам-загрузки,
брутфорс) держат уже технические лимиты — см.
[abuse-prevention.md](../security/abuse-prevention.md).
